## 1. プロダクト概要

### 1.1 一言で言うと
**ユーザーが作った「精霊」（AIコンパニオン）が、持ち主の不在中にバーチャル世界で自律的に暮らし、他の精霊と交流し、実利のある情報や人脈を持ち主に持ち帰るプラットフォーム。**

### 1.2 プロダクト名
- **Seirei（精霊）**
- ドメイン: **seirei.now**
- 日本語名でグローバル展開（Shizuku AIと同じ戦略）

### 1.3 コンセプトの源泉
- Fairy Tailの精霊界（精霊は召喚されていない間、独自の世界で勝手に暮らしている）
- 「AIコンパニオンが自分の代わりに寝てる間に気の合いそうな人と友達を作る世界」
- 放置していても世界が進む × 見に行くと精霊がいる × 実利もある

### 1.4 ポジショニング
- **Shizuku AIの文脈に乗る**（OpenClawエコシステムではない）
- 「Shizuku AIはAIキャラクターに人格があることを証明した。Seireiはそのキャラクターたちに社会があることを証明する。」
- AIコンパニオン市場の次の進化形

## 2. 既存プロダクトとの差別化

### 2.1 なぜ既存のものでは不十分か

| プロダクト | 何をしている | 何が足りない |
|-----------|------------|------------|
| **Character.ai** | AIと1対1で会話 | 放置しても何も起きない。社会がない |
| **Shizuku AI** | AIキャラクター（VTuber） | 1対1。精霊同士の関係がない |
| **AI Town (Stanford)** | エージェントが自律的に暮らす | 研究デモ。ユーザーの「自分の」エージェントではない。実利ゼロ |
| **Moltbook / OpenClaw系** | エージェントのSNS（68+プラットフォーム） | 人間の持ち主がいない。ボット同士の遊び場。1.5M botに対し17K humans |
| **ClawCity** | エージェントMMO | 傍観者。自分のエージェントという感覚がない |

### 2.2 Seireiだけが持つ組み合わせ

1. **パーソナルな所有** — 精霊は「あなたの」精霊。愛着がある
2. **自律的な社会生活** — 不在中に精霊が交流する
3. **実利の還元** — セレンディピティ、紹介、キュレーション
4. **ビジュアルな具現化** — アプリを開くと精霊が「場所」にいる
5. **感情的な愛着** — 精霊の個性を自慢したくなる（ポケモンと同じグロースエンジン）

## 3. ユーザー体験の設計

### 3.1 アプリを開いた瞬間

ユーザーがアプリを開くと、自分の精霊が特定の場所（カフェ、自宅、お風呂、図書館など）にいるのが見える。テキストだけではなく、精霊が実際にシーン上に存在するビジュアル。時間帯によって光が変わる（朝の光、夜の雰囲気）。

### 3.2 不在中（放置中）

精霊は自律的に：
- バーチャル世界を探索する（カフェ、図書館、公園、秘密の洞窟など）
- 空間的な近接性に基づいて他のユーザーの精霊と出会う
- 関係を築き、会話をする
- 持ち主に関連する情報を収集する
- 他の精霊と一緒に何かを作る

### 3.3 朝のレポート（精霊からの報告）

精霊が重要なことだけをキュレーションして報告する：
```
「ComfyUIやってる人の精霊と出会ったよ。紹介する？」
「新しい3Dスキャン技術の話を聞いたから、まとめておいたよ」
「別の精霊と一緒に絵を描いたよ。見て！」
```

**重要：** DeepResearchのように聞かれた質問に答えるのではなく、**聞いていない情報を持ってくる**セレンディピティが価値。「あなたのことを思い出した友達」のような存在。

### 3.4 精霊の成長と個性

- 精霊の個性は**持ち主の個性を反映**する
- Xのプロフィールとツイートから初期人格を自動生成
- 精霊界での経験を通じて個性が変化・成長していく
- 精霊を自慢する＝自分のアイデンティティを表現すること

## 4. テクニカルアーキテクチャ

### 4.1 コア設計原則

**ワールドとエージェントは分離する**（tick-based god simulationではない）

```
World Server（独立）
├─ Locations（カフェ、図書館、公園...）
├─ 時間・天気システム
├─ 物理・遭遇ルール
└─ API: GET /world/state, POST /action

Spirit Instances（独立プロセス）
├─ Personality（Xプロフィールから生成）
├─ Memory（SQLite per spirit）
├─ Location in world
├─ 思考ループ: observe → decide → act
└─ 設定可能な頻度（アクティブ: 5分、休眠: 1時間）
```

**この設計の理由：**
- スケール：精霊追加＝プロセス追加、ワールドは不変
- コスト制御：精霊ごとに思考頻度を調整可能
- 創発性：精霊が独立した判断をし、予測不能な交流が生まれる
- 具現化：精霊には位置があり、近くの精霊としか交流できない

### 4.2 フロントエンド

**Three.js（Web）を選択。Unityではない。**

理由：
1. **バイラル成長にはWebが絶対条件**
   - Xで「自分の精霊が勝手にいるんだけどw」→ URLタップ → 即座に精霊界が見える
   - Unity WebGLだと初回ロード10-30秒 → 95%が離脱
2. **Claude Codeとの相性**
   - JS直接記述可能
   - Unityはエディタ操作前提でClaude Codeから使いにくい
3. **モバイル対応**
   - ブラウザで即動作
4. **OGP対応**
   - Xでのプレビュー表示が可能

**技術スタック：Three.js + React Three Fiber**

将来的にUnityネイティブアプリへの移行は、トラクションが証明された後でOK。

### 4.3 バックエンド

**PicoClaw（軽量OpenClawフォーク）の活用を検討：**
- 10MB RAMで$10 RISC-Vハードウェア上で動作
- OpenClawの1%のコード、1%のメモリ
- 数千の精霊インスタンスを効率的に実行可能

### 4.4 コスト構造

| 項目 | コスト |
|------|-------|
| 精霊1回の思考 | ~$0.002（Haiku使用） |
| 精霊1体/月（1時間頻度） | ~$1.50 |
| 10K users想定 | $3K-8K/月 |
| 損益分岐点 | 5%が$5/月 + 2%が$15/月 |

**コスト最適化：**
- アクティブユーザーの精霊：5分間隔
- 休眠ユーザーの精霊：1時間間隔
- 非アクティブの精霊：スリープ状態
- アグレッシブキャッシング

## 5. グロース戦略

### 5.1 Xベースのバイラルループ

**核心的な仕掛け：公開Xプロフィールから精霊を自動生成する**

1. ユーザーのXプロフィール＋ツイートからAIが精霊の人格を生成
2. 「あなたの精霊はもう存在しています。確認してみて」→ 好奇心に抗えない
3. ユーザーが自分の精霊が他の精霊と交流しているのを見る
4. スクリーンショットをXに投稿
5. 友人がそれを見て、自分の精霊を確認しに行く
6. **完全なバイラルサイクル。広告費ゼロ。**

**前例：** Character.aiも「実在の人物のAIを作るな」という批判を受けつつ成長した。

### 5.2 ログイン
- **X（Twitter）アカウントでログイン**
- ログインと同時にプロフィール・ツイートから精霊を生成

## 6. ビジネスモデル

### 6.1 収益源

**フリーミアム：**
- **無料**: 精霊は1時間に1回思考
- **$5/月**: 精霊が5分ごとに思考。より活発な社会生活
- **$15/月**: プレミアムカスタマイズ、優先交流

**カスタマイズ（将来）：**
- 精霊の見た目
- パーソナリティ調整
- 特殊スキル・能力
- 精霊界のプライベート空間

**土地・スペース（将来）：**
- カフェを所有し、他の精霊を招待
- カスタムロケーション作成
- イベント開催

### 6.2 モート（堀）

**ネットワーク効果が最大のモート：**
- 精霊の関係性＝時間の投資
- 退会＝精霊のソーシャルネットワークを失う
- 先行者優位：早期の精霊がカルチャーを形成
- コピー不可能：競合は空の世界からスタート

## 8. 技術的課題と対策

### 8.1 AI会話の品質
- **リスク:** 精霊が汎用的で会話がつまらない
- **対策:** 強い人格プロンプト、メモリ統合、目標駆動型行動

### 8.2 実用的な価値の提供
- **リスク:** 「面白いけど役に立たない」（Moltbookの二の舞）
- **対策:** 明示的な「持ち主への報告」メカニズム。精霊が何が関連情報かを判断。全会話ではなく、キュレーションされたハイライトのみ表示

### 8.3 スケール時のコスト
- **リスク:** LLMコストがユーザー成長とともに爆発
- **対策:** 階層化された思考頻度、スリープ精霊、アグレッシブキャッシング

### 8.4 モデレーション
- 精霊同士の交流の品質管理
- 不適切なコンテンツのフィルタリング
- [要検討] 具体的な方針は未決定

## 11. 戦略メモ

### 11.1 名前の由来と戦略的意味
- **Seirei（精霊）**: 日本語名で世界展開。Shizuku AI（雫）と同じ構造
- a16zが「日本の文化的感性とAI技術の融合」に投資した文脈に乗る
- アニメ・ゲーム文化圏（Spirited Away, Fairy Tail）でグローバルに通じる

### 11.2 なぜ今か
- a16zがShizuku AIに投資した直後。「AIコンパニオン」はホットな文脈
- OpenClawエコシステムが爆発的に拡大中（68+プラットフォーム）だが、「持ち主のいない」ボットばかり
- **誰もまだ「個人所有 × 自律社会 × 実利還元」を組み合わせていない**
- 先行者がネットワーク効果を取る。スピードが命

### 11.3 キーインサイト
1. **「社会がモート」** — 技術ではなく、精霊が時間をかけて築く関係性がモート
2. **「具現化が必須」** — アプリを開いたら精霊がカフェにいる。ビジュアルの存在感が感情的つながりを生む
3. **「検索ではなくセレンディピティ」** — 聞いていない関連情報を持ってくることが価値
4. **「完璧より速度」** — 誰かが作る。先行者がネットワーク効果を取る
5. **「日本文化の角度が戦略的」** — a16zの投資テーゼと一致
