# 精霊行動システム設計

> 作成日: 2026-02-11
> ステータス: 一部実装済み

---

## 1. 現状の問題

現在の精霊は15秒固定tickでLLMを呼び出し、毎回「何かしろ」と強制される。

| 問題 | 具体例 |
|---|---|
| 不自然なテンポ | 会話中でも15秒待つ。景色を眺めるだけでいい場面でも動く |
| LLMの無駄遣い | 何もない場所でobserveし続ける |
| 意志がない | 「次何する？」を毎回聞かれるだけ。目的を持って行動していない |
| コスト制御が外部的 | tick間隔でしかLLMコストを制限できない |

---

## 2. 設計方針

**固定tickを廃止し、状態 + 目的 + リソースに基づく行動ループに置き換える。**

精霊は「何を達成したいか（goal）」を持ち、状況に応じてアプローチ（subgoal）を選び、具体的な行動（ツール呼び出し）はLLMが毎回判断する。

体力と思考力がリソースとして行動を自然に制限し、これがそのまま課金・レベルシステムに接続する。

---

## 3. 精霊の内部モデル

```
Spirit
├── state: idle | active | conversing | resting
├── goal: "友達を作りたい"          ← 高レベルな目的
│   └── subgoal: "みんなに話を聞く"  ← 現在のアプローチ
├── stamina: 80/100                ← 体力（歩行で消費、休息で回復）
├── mentalEnergy: 60/100           ← 思考力（LLM呼び出しで消費、時間で回復）
├── level: 3                       ← 上限・回復速度に影響
└── tier: free | premium           ← 課金で上限UP・回復速度UP
```

### goal と subgoal

| 概念 | 粒度 | 例 | 誰が決める |
|---|---|---|---|
| goal | 大きな目的 | 友達を作りたい / ワールドを探索したい / 静かに過ごしたい | LLM（長期） |
| subgoal | 今のアプローチ | みんなに話を聞く / 噴水の周りを散歩する / 木の下で休む | LLM（中期） |
| action | 具体的行動 | walk_to, look_at, say, observe | LLM（毎ターン判断） |

**actionをsubgoalに含めない。** subgoalは方向性であり、スクリプトではない。同じsubgoal「みんなに話を聞く」でも状況によって行動が変わる:

- 精霊が見えた → 近づいて話しかける
- 誰もいない → 別の場所に移動して探す
- 体力が低い → 今いる場所で声を出して呼ぶ

## 4. 状態（state）

### 状態一覧

| state | 説明 | 思考間隔 | stamina消費 |
|---|---|---|---|
| **idle** | 何もしていない。ぼーっとしている | 17-27秒 | 回復中 |
| **active** | goalに向かって行動中 | 6-9秒 | 行動による |
| **conversing** | 他の精霊と会話中 | 2-3秒 | 低 |
| **resting** | 体力/思考力回復中 | 6-9秒(チェック間隔) | 急速回復 |

### 状態遷移

```
         ┌─────────────────────────────┐
         │         idle                │
         │   (目的なし、ぼんやり)        │
         └──┬──────────┬───────────┬───┘
            │          │           │
    goalが生まれた  声が聞こえた  stamina低下
            │          │           │
            ▼          ▼           ▼
       ┌────────┐ ┌──────────┐ ┌─────────┐
       │ active │ │conversing│ │ resting │
       │(行動中)│ │ (会話中)  │ │ (休憩中) │
       └───┬────┘ └────┬─────┘ └────┬────┘
           │           │            │
     goal達成/断念  会話終了    stamina回復
           │           │            │
           └───────────┴────────────┘
                       │
                       ▼
                     idle
```

### イベント割り込み

どの状態でも、以下のイベントで即座に状態遷移できる:

| イベント | 発火条件 | 遷移先 |
|---|---|---|
| 声が聞こえた（自分宛） | say()の到達判定 | conversing（即応答） |
| 声が聞こえた（他者宛/独り言） | say()の到達判定 | LLMが判断（割り込むか無視するか） |
| stamina枯渇 | stamina ≤ 0 | resting（強制） |
| mentalEnergy枯渇 | mentalEnergy ≤ 0 | resting（思考停止） |

## 5. リソースシステム

### stamina（体力）

| 行動 | 消費量 |
|---|---|
| walk_to / move_to | 移動距離 × 1.0 |
| say (shout) | 3 |
| say (normal) | 1 |
| idle / resting | **回復**: 2/分 |

### mentalEnergy（思考力）

| 行動 | 消費量 |
|---|---|
| LLM呼び出し（1回） | 5-10（応答の長さによる） |
| イベント反応 | 3（短い判断） |
| goal生成 | 15（深い思考） |
| resting | **回復**: 3/分 × 3.0(rest倍率) = 9/分 |

### レベルによる変化

| レベル | stamina上限 | mentalEnergy上限 | 回復速度倍率 |
|---|---|---|---|
| 1 | 200 | 100 | 1.0x |
| 5 | 100 | 60 | 1.2x |
| 10 | 120 | 80 | 1.5x |
| 20 | 150 | 100 | 2.0x |

### 課金ティアによる変化

| | free | premium ($5/月) | premium+ ($15/月) |
|---|---|---|---|
| mentalEnergy上限倍率 | 1.0x | 2.0x | 4.0x |
| mentalEnergy回復速度 | 3/分 | (未実装) | (未実装) |
| stamina上限倍率 | 1.0x | 1.0x | 1.5x |

**結果**: freeの精霊はmentalEnergy 100で、1回5-10消費 → 10-20回思考したら回復待ち。premiumなら80で → 倍の活動量。これがそのままLLM APIコスト制御になる。

## 6. 行動ループ（新設計）

```
loop:
  1. イベントキューを確認
     └─ 声が届いている？ → 即反応（conversingへ）

  2. 現在のstateに応じた処理
     ├─ resting → stamina/mentalEnergy回復待ち → idle へ
     ├─ conversing → 短い間隔でLLM呼び出し → 会話が途切れたらactiveかidleへ
     ├─ active → subgoalに沿ってLLM呼び出し → goal達成/断念でidleへ
     └─ idle → 長い間隔 → mentalEnergy十分ならLLMにgoal生成を依頼 → activeへ

  3. リソースチェック
     ├─ stamina ≤ 0 → resting強制
     ├─ mentalEnergy ≤ 0 → resting強制
     └─ OK → 継続

  4. 次の思考タイミングを計算してsleep
```

### LLMに渡すコンテキスト

```
あなたの状態: active
目標: 友達を作りたい
アプローチ: みんなに話を聞く
体力: 80/100  思考力: 55/100

【観察結果】時間帯: evening
見えるもの:
  - fountain-0 (fountain): 距離3.2
近くの精霊:
  - Kaze (ID: spirit-go-2): 距離4.1, 位置[-2.0, 3.5]
聞こえた声: なし

→ 次の行動を決めてください。
  体力や思考力が低いときは無理せず休んでもOKです。
  目標やアプローチを変えたいときは宣言してください。
```

### LLMの応答に含まれうるもの

- ツール呼び出し（action）: walk_to, look_at, say, observe
- 目標変更: 「目標を変更: ワールドを探索したい」
- アプローチ変更: 「アプローチを変更: 一人で噴水を見に行く」
- 休憩宣言: 「少し休みたい」→ resting へ

## 7. イベントシステム

### 現状

Go精霊 → HTTPポーリング → TSサーバー。サーバーからGoへのプッシュ手段がない。

### 選択肢

| 方式 | 実装量 | リアルタイム性 | 備考 |
|---|---|---|---|
| **短間隔ポーリング** | 小 | 1-2秒遅延 | conversingステートの時だけ高頻度（2秒）でobserve |
| **SSE (Server-Sent Events)** | 中 | 即時 | サーバー→Go単方向。イベント通知に最適 |
| **WebSocket** | 大 | 即時 | 双方向だがオーバーキル |

**推奨: まず短間隔ポーリングで実装し、必要ならSSEに移行。**

conversingステートの2秒ポーリングで十分な会話テンポが得られる。idle時は30秒ポーリングでイベント確認。

### イベントの流れ（ポーリング方式）

```
Hikari(idle, 30秒間隔) ─── observe ──→ voices: []   （何もない）
                                         ... 30秒後 ...
Kaze ──── say("おーい！", shout) ──→ サーバーに保存
Hikari(idle, 30秒間隔) ─── observe ──→ voices: [{from: "Kaze", message: "おーい！"}]
                            ↓
                    state → conversing（3秒間隔に切替）
Hikari ─── look_at(Kaze) + say("はーい！") ──→
                            ... 3秒後 ...
Hikari ─── observe ──→ voices: [{from: "Kaze", ...}]
                            ... 会話継続 ...
                            ... 30秒声なし ...
                    state → idle（30秒間隔に戻る）
```

## 9. ThinkサイクルとTickの関係

### 構造

精霊の行動ループは1秒ごとのtickで回る。各tickで世界をObserveし、条件を満たせばThink（`ProcessDirect`）を呼ぶ。

```
世界のタイムライン（1秒tick）
tick 1   tick 2   tick 3   ...   tick 18  tick 19

精霊のgoroutine:
  ├── Observe → 状態チェック → shouldThink=true
  ├── THINK開始 ─── ProcessDirect() ブロック中 ──────────┐
  │   │  [PicoClaw内部ループ iteration 1~maxIterations]  │
  │   │   LLM呼出 → tool_call → 実行 → LLM呼出 → ...    │
  │   │                       例: 17.9秒経過             │
  ├── THINK終了 ←────────────────────────────────────────┘
  ├── 結果処理（REPLY出力、ME消費、lastThink更新）
  ├── sleep(1秒) → 次のtick
  ├── Observe → shouldThink=false（まだ間隔に達していない）
  ├── sleep(1秒)
  └── （shouldThink間隔が経過するまでObserveだけ繰り返す）
```

**Think中は精霊は世界を知覚できない（盲目状態）。** 世界サーバーや他の精霊は独立して動き続ける。

### 重要なパラメータ関係: maxIterations × shouldThink間隔

`maxIterations`（1回のThink内のLLM↔ツール往復上限）と `shouldThink間隔`（Think間の待機時間）はセットで調整する必要がある。

```
現在:  maxIter=4, 間隔=17-27秒(idle) → API呼出レートは状態依存
変更例: maxIter=4, 間隔=20秒 → 20秒ごとに最大4回API呼出 = 最大0.2回/秒
```

**両方を半分にすればAPI呼び出しレートは同じまま、世界の観測頻度が2倍になる。**

| パラメータ | 大きい | 小さい |
|---|---|---|
| **maxIterations** | 1回で複雑な行動が可能（goal→移動→観察→会話）、長時間盲目 | 行動は1-2手ずつ、頻繁に世界を知覚、リアクティブ |
| **shouldThink間隔** | API呼び出し頻度が低い、世界の変化への反応が遅い | 頻繁に思考、変化に素早く反応、APIコスト増 |

### 設計原則

- maxIterationsを下げてThinkを短く・頻繁にするほうが、精霊の「世界への応答性」が上がる
- iteration間にdelayを入れるアプローチは無意味（盲目のまま行動が遅くなるだけ）
- 精霊の個性としてこれらのパラメータを変えることも可能（おっとりした精霊=長Think、せっかちな精霊=短Think）

### ProcessDirectの終了条件

| 条件 | 意味 |
|---|---|
| LLMがツールなしのテキスト応答を返した | 正常終了（精霊が「考え終わった」） |
| iteration >= maxIterations | 強制終了（精霊が「考え込みすぎた」→ 応答なし） |
