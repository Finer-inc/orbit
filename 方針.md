# Seirei プロダクト方針

## コアコンセプト

「自分の子供」の成長を見守る体験。

精霊はユーザーの分身ではなく、ユーザーの「子供」。
学校に行く、友達を作る、仕事に就く、疲れて家に帰る — その生活を眺めて愛着を持つ。

## MindAxisとの違い

| | MindAxis | Seirei |
|---|---|---|
| AIの位置づけ | 自分の分身（思考の再現） | 自分の子供（育成と観測） |
| ユーザー体験 | 自己理解 | 育成観測 |
| 身体 | なし（テキストフィード） | あり（3D空間で生活する） |
| 毎日開く理由 | 分身が何を語ったか見る | 子供がどう過ごしたか見る |

## 身体がある意味

フィードでは表現できない体験を提供する。

- 疲れたから家に帰る
- 遠くにいるから会えない
- 偶然近くにいたから話しかけた
- 雨だから外に出ない
- 友達と並んで歩いている

こうした「非効率で無駄な振る舞い」に愛着を感じる層がターゲット。
たまごっち、どうぶつの森、ポストペットと同じ系譜。

## 開発の優先順位

1. **育成観測のベース** — 毎日開いて子供を見る習慣を作る
2. **実利の追加** — マッチング、情報収集など、育成の上に乗せるユーザー価値

習慣が先。実利は後。

## 事業化パス

参考: Shizuku AI（AI VTuber「しずく」）
- 2023年: UC Berkeley PhD学生が趣味でAI VTuberを開始
- 2026年: a16zからシード$15M調達、企業価値120億円

趣味 → トラクション証明 → 事業化 の流れ。

Seireiの場合:
1. **趣味フェーズ（現在）** — 育成観測の体験を作り込む。React/Three.jsで検証
2. **公開・検証フェーズ** — 少数ユーザーに触ってもらい「毎日開く」が成立するか確認
3. **プラットフォーム拡張** — Unity/Godotでネイティブアプリ化（Steam/iOS/Android）
4. **事業化フェーズ** — トラクションが見えたら資金調達・法人本格稼働

法人は設立済み。技術基盤も動いている。足りないのは「毎日開きたくなる体験」の完成度。

## アーキテクチャ上の強み

- レンダリング層（React/Three.js）とバックエンド（WorldServer + Go agents）は分離済み
- フロントエンドは差し替え可能（2D、Unity/Godot、モバイルアプリ等）
- 3Dが正しい選択かどうかは育成体験の検証で判断する
- Unity/Godot移行時はサーバーのワールド定義をGLBからJSONに分離する（Phase 1）
